## Architecture

Schema markup on this site is delivered as `<script type="application/ld+json">` blocks in the `<head>` of every page. There are two systems generating schema, both running server-side at build time. No JavaScript execution required. AI crawlers see the schema on first parse.

### System 1: Blowfish Theme (automatic)

**File:** `themes/blowfish/layouts/partials/schema.html`

Do not edit this file. It is part of the theme submodule.

This partial runs on every page automatically. It generates:

**Homepage only:**
- `WebSite` schema with site title, description, language, URL, publisher

**Every other page:**
- `Article` schema with title, headline, description, author, dates, keywords, word count
- `BreadcrumbList` schema with the full ancestor path (enabled by `enableStructuredBreadcrumbs = true` in `params.toml`)

Blowfish pulls data from:
- `config/_default/languages.en.toml` (site title, author name, description)
- Page frontmatter (title, description, date, tags, publishDate, lastmod)
- Hugo built-in fields (Permalink, WordCount, Section, Ancestors)

### System 2: Custom Partial (convention-based)

**File:** `layouts/partials/extend-head-uncached.html`

This is our custom partial. It hooks into Blowfish via the `extend-head-uncached` injection point. Blowfish calls it on every page with full page context. It is NOT cached, meaning it runs per page and can output different schema for each page.

The other injection point, `extend-head.html`, is cached (runs once per build with only site context). That one is used for font preconnects and cannot do per-page schema.

This partial generates schema based on URL path detection. No frontmatter flags required.

## Schema Types by Page

| Page | Blowfish Generates | Custom Partial Generates |
|---|---|---|
| Homepage (`/`) | `WebSite` | `Organization` + `ProfessionalService` |
| Service pages (`/services/*`) | `Article` + `BreadcrumbList` | `Service` |
| FAQ pages (`/research/quick-faqs/*`) | `Article` + `BreadcrumbList` | `FAQPage` |
| Everything else | `Article` + `BreadcrumbList` | Nothing |

Multiple JSON-LD blocks on one page is valid and expected. Google and AI crawlers merge them.

## What Each Type Contains

### Organization (homepage only)

- Company name, URL, logo
- Social links (LinkedIn, Twitter, Instagram, GitHub) via `sameAs`
- `@id` anchor: `https://digitalcrunch.com/#organization`

The `@id` anchor lets other schema blocks reference this entity without repeating all the data. Service pages reference it in their `provider` field.

### ProfessionalService (homepage only)

- Full NAP: name, address, phone, email
- Business hours (Mon-Fri 9-5, Sat 8-1)
- Area served (United States)
- Price range (`$$`)
- Founder (James Fraze, Cybersecurity Engineer)
- `@id` anchor: `https://digitalcrunch.com/#profservice`

### Service (any page under /services/)

Detection: URL contains `/services/` and is not the `/services/` index page itself. Works for nested pages like `/services/ai-bots/ai-voice-receptionist/`.

Data pulled automatically from existing frontmatter:
- `title` becomes `name`
- `description` becomes `description`
- `Permalink` becomes `url`
- `Lastmod` becomes `dateModified`
- Provider references the Organization entity via `@id`

Optional frontmatter overrides (add these to a service page only when needed):

```yaml
schema:
  serviceType: "AI Voice Agent Implementation"
  priceRange: "$$$"
```

If the `schema` block is absent, the Service schema still generates correctly from standard frontmatter fields.

### FAQPage (any page under /research/quick-faqs/)

Detection: URL contains `/quick-faqs/`.

The partial parses the page's raw markdown content:
1. Splits on `####` (h4 headings)
2. First line after `####` becomes the question name
3. All non-empty lines after the first blank line become the answer text, concatenated with spaces
4. Markdown formatting is stripped from answers: `**bold**` becomes `bold`, `[link text](url)` becomes `link text`, list markers (`- `) are removed

This means FAQ schema updates automatically when you edit questions. No separate data source to maintain.

## File Locations

| File | Purpose | Edit? |
|---|---|---|
| `themes/blowfish/layouts/partials/schema.html` | Blowfish Article/WebSite/BreadcrumbList | Never. Theme submodule. |
| `layouts/partials/extend-head-uncached.html` | Our Organization/ProfessionalService/Service/FAQPage | Yes. This is our file. |
| `layouts/partials/extend-head.html` | Font preconnects (cached, site-level) | Not for schema. |
| `layouts/partials/footer.html` | Footer with Microdata ProfessionalService | Separate from JSON-LD schema. |
| `config/_default/params.toml` | `enableStructuredBreadcrumbs = true` | Controls BreadcrumbList generation. |

### Footer Microdata

The footer (`layouts/partials/footer.html`) contains inline Microdata markup (`itemscope itemtype="https://schema.org/ProfessionalService"`) for the visible NAP block. This is a separate implementation from the JSON-LD in the head. Both are valid. The Microdata is tied to the visual content. The JSON-LD is machine-readable data in the head.

## How to Add a New Service Page

1. Create the page under `content/services/` with standard frontmatter (`title`, `description`, `date`)
2. Build. Service schema generates automatically.
3. Optionally add `schema.serviceType` and `schema.priceRange` to frontmatter for richer data.
4. Validate with Google Rich Results Test.

No template edits required.

## How to Add a New FAQ Page

1. Create the page under `content/research/quick-faqs/` with `_index.md`
2. Use `####` headings for questions, followed by a blank line, then the answer paragraph(s)
3. Build. FAQPage schema generates automatically with all Q&A pairs parsed.
4. Validate with Google Rich Results Test.

No template edits required. The partial parses the content structure.

## Known Behaviors

### Hugo JS Context Escaping

Hugo's template engine applies JavaScript context escaping inside `<script>` tags. This produces:
- `/` becomes `\/` in template variables (not in hardcoded strings)
- `+` becomes `\u002b`
- `&` becomes `\u0026`

These are all valid JSON per RFC 7159. Every JSON parser and every schema validator handles them correctly. This is cosmetic only. Blowfish's own schema.html avoids this by using `safeURL` without quotes and letting Hugo add the quotes. Our partial uses `safeJS` inside quoted strings, which does not fully bypass the escaping.

### FAQ Answer Parsing

The FAQ parser strips basic markdown but does not handle all edge cases:
- `**bold**` and `*italic*` are stripped
- `[link text](url)` becomes `link text`
- `- ` list markers are removed
- Inline code backticks are not stripped
- Nested markdown (bold inside a link) may not fully clean

For best schema output, write FAQ answers as clean prose without heavy formatting.

### Blowfish Article Schema on Service Pages

Blowfish generates `Article` schema on service pages because it treats all non-homepage content as articles. This is technically incorrect (a service page is not an article) but does not cause validation errors. Google uses the most specific schema type when multiple types exist. Our `Service` schema is more specific and takes precedence.

## Testing and Validation

### Validators

- **Google Rich Results Test:** [https://search.google.com/test/rich-results](https://search.google.com/test/rich-results) -- Tests live URL or pasted HTML. Shows which rich result types are detected. This is the validator that matters for Google Search.
- **Schema.org Validator:** [https://validator.schema.org](https://validator.schema.org) -- Tests against the full Schema.org vocabulary. More strict than Google's tool. Good for catching structural errors.

### Local Testing

After editing the partial or adding new content:

```bash
hugo --quiet
```

Then inspect the generated HTML:

```bash
# Count JSON-LD blocks on a page
grep -c 'ld+json' public/index.html

# View the actual schema output
grep -n 'script type="application/ld+json"' public/services/ai-automation/index.html

# Read the schema block (find the line number from grep, then read from there)
sed -n '307,325p' public/services/ai-automation/index.html

# Count FAQ questions parsed
grep -c '"@type": "Question"' public/research/quick-faqs/voice-receptionist/index.html
```

### Expected Block Counts

| Page | JSON-LD Blocks |
|---|---|
| Homepage | 3 (WebSite + Organization + ProfessionalService) |
| Service page | 2 (Article + Service) |
| FAQ page | 2 (Article + FAQPage) |
| Research/other | 1 (Article with BreadcrumbList inside) |

### What to Check After Changes

1. Build completes without errors
2. Correct number of JSON-LD blocks per page type
3. Service pages show `"@type": "Service"` with correct title and description
4. FAQ pages show all questions parsed (count `"@type": "Question"` occurrences)
5. Homepage shows Organization with social links and ProfessionalService with full NAP
6. Non-target pages (research articles, about pages) have no extra schema blocks
7. Paste a live URL into Google Rich Results Test and confirm detection

## Extending the Schema

### Adding a New Schema Type

To add schema for a new content section (e.g., `HowTo` for tutorial pages):

1. Open `layouts/partials/extend-head-uncached.html`
2. Add a new `else if` block before the final `end`:

```
{{- else if in .RelPermalink "/your-section/" -}}
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "HowTo",
  "name": "{{ .Title | safeJS }}",
  ...
}
</script>
```

3. Build and validate.

### Updating Business Data

Business name, address, phone, hours, and social links are hardcoded in the custom partial. If any of these change, edit `layouts/partials/extend-head-uncached.html` directly. Search for the value you need to change. It appears in both the Organization and ProfessionalService blocks on the homepage section.

The footer Microdata in `layouts/partials/footer.html` has its own copy of NAP data. Update both files if business details change.
